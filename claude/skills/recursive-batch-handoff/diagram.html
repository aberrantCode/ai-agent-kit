Recursive Batch Handoff

no-master-plan pattern · discovery-driven batching · self-similar handoff prompt

Purpose
Use when a large migration, refactor, or long-running batch operation must be split across many sessions or iterations — each batch runs a discovery command to find what's left, executes one coherent chunk, and emits a self-similar handoff prompt so the next session (or run) continues without a central master plan.
Large file migrations
Dozens to hundreds of homogeneous targets (e.g. ~200 scripts) that cannot fit in one session or one PR.
Live/queued operations
ffmpeg jobs, DB migrations, staged deployments that must proceed in controlled increments.
Multi-phase infra/feature work
Spans many sessions, where a centralized master plan would go stale or drift out of sync.
eight-step method, one recursive loop
Method · re-run every session
  • 1
    Discovery command = source of truth
    grep/query/count for "what's left." Re-run fresh at the start of every session — never trust a stale summary.
  • 2
    Select next batch
    Coherent grouping heuristic — next largest prefix group, subdirectory, or glob. Cap size (~20-30 files) to stay under an ~800-line diff.
  • 3
    Execute with judgment-per-file
    Not blind sed/find-replace — similar files can still need individual handling. Normal TDD discipline applies.
  • 4
    Commit the batch
    One PR / one change per batch.
  • 5
    Update progress log
    Prepend a new entry to docs/progress.md so the most recent state sits at the top — survey the whole sequence and spot stalls at a glance.
  • 6
    Log recurring failure modes
    e.g. "already-fixed-in-code stale backlog rows" — logged at the top of the handoff so the next session doesn't rediscover it.
  • 7
    Emit self-similar handoff prompt
    Same template, updated state, run verbatim next time. Embeds the discovery command, batch heuristic, execution recipe, and an instruction to repeat + re-emit itself.
  • 8
    Live/queued: verify drain before scaling
    Confirm the prior batch's count is falling before increasing batch size — only scale up once it has fully cleared.
discover→ select→ execute→ commit→ log→ handoff→ repeat
Branch A · File/refactor batches
Fixed-scope targets
Files/scripts/configs. Batch size is capped by diff size (~20-30 files, ~800 lines) — cap stays constant across sessions.
Risk to guard
Treating a "coherent" batch as uniform and applying blind scripted replacement across it.
Branch B · Live/queued operations
In-flight targets
ffmpeg jobs, DB migrations, deployments. Batch size is dynamic — grows only after the prior batch is confirmed draining.
Risk to guard
Scaling up before the previous batch fully clears risks orphaning a queue across a service restart.
Inputs → Outputs · per iteration
Inputs
Fresh discovery-command output · the prior handoff prompt (as the recipe, not as scope truth) · docs/progress.md history.
Outputs
One committed batch/PR · a prepended docs/progress.md entry · a self-similar handoff prompt ready to paste into the next session.
Gotchas
  • Skipping discovery and trusting a prior session's written summary lets scope drift out of sync with reality.
  • Blind scripted replacement (sed, mass find/replace) across a "coherent" batch causes regressions — similar files still need per-file judgment.
  • Omitting the recurring-failure-mode note means every future session rediscovers the same gotcha independently.
  • Batches over the size/line cap block review and reintroduce the exact master-plan bottleneck this pattern avoids.
  • For live/queued operations, scaling batch size before confirming the previous batch drained risks orphaning in-flight work.
  • Never run a separate master plan alongside this — the progress log and discovery command must stay the sole source of truth (proven to ~200 files across 15+ sessions).
