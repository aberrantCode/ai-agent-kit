Diagnostics Probe Design

verify premise → map boundaries → read-only multi-hypothesis probe → symptom-indexed registry

Purpose
Before proposing any fix to a broken service or infrastructure, verify the premise, then write a read-only, multi-hypothesis probe that combines recorded metrics with live state to pinpoint the broken component — avoiding redoing already-completed or destructive work.
Service reported down
A monitor or integration is broken and you're about to investigate root cause.
"Needs retiring/building"
A prompt claims infra needs action — check before acting on the claim.
Recurring "fixed" issue
A previously fixed issue recurs, or triage keeps reinventing the same steps.
nine-step method, evidence before conclusions
Method
  • 1
    Verify the premise first
    30-second check: read docs/progress.md, glob for the artifact, run git log --oneline --all. Prevents redoing destructive work.
  • 2
    Map component boundaries
    Enumerate each hop in the chain and what data enters/exits it. The break is almost always at a boundary.
  • 3
    Write the probe before conclusions
    Read-only, multi-hypothesis, numbered sections each testing one hypothesis, ending in a decision tree.
  • 4
    Gather two evidence classes
    Recorded metrics (what happened) + live state (what's true now) — per hypothesis.
  • 5
    Pre-validate the substrate
    Probe the Prometheus/Loki scraper to confirm exact metric names before writing PromQL/LogQL. Skipping causes ~60% of follow-up fixes.
  • 6
    Capture errors verbatim
    Exact strings and timestamps, not paraphrased — the literal message is often the signal.
  • 7
    Bisect if unresolved
    Change one variable at a time with throwaway targeted tests to pin the exact trigger.
  • 8
    Register by symptom, not internals
    Index by what the operator observes; store the exact cross-host invocation, not a bare script path.
  • 9
    Document the decision tree
    Include false-positive misdiagnoses so the next investigator lands on root cause directly.
Two Evidence Classes · use both, not one as substitute
Recorded metrics
Prometheus/Loki time-series scoped to the incident window. Immutable and reproducible — establishes what actually happened.
Live state
cgroup files, process state, API queries (Authentik, DNS). Confirms what's true right now.
Gotchas
  • Don't skip reproduction or assume root cause before evidence pinpoints the failing boundary.
  • Don't act on a prompt's claim something "needs" retiring/building without checking progress logs and git history first.
  • Don't index a probe registry by probe/script internals — operators search by symptom.
  • Don't store bare script paths for cross-host checks — multi-hop ssh invocations get lost and each triage reinvents them.
  • Don't write PromQL/LogQL against assumed metric names — exporter drift silently breaks queries.
  • Don't paraphrase captured error messages — verbatim text (with timestamps) preserves the signal for bisection.
