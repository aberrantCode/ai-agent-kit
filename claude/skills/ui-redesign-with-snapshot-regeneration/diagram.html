UI Redesign with Snapshot Regeneration

split data from presentation · regenerate deliberately · verify with a clean second pass

When to Use
Any multi-phase UI redesign/reskin with existing snapshot/visual tests — especially where the framework has behavioral constraints (e.g. React 19's no-setState-in-effect rule) that make silent regressions easy. Also any time you're about to run vitest -u after a deliberate visual change, since regeneration can hide non-snapshot failures if done carelessly.
seven-step method
Method
  • Separate data layer from presentation layer
    Extract actions/hooks/query keys first, mount new component trees on the same shared state. "Broke behavior" vs. "changed appearance" become separable questions.
  • Stage phases on feature branches → dev
    Each phase gated by tsc, eslint, vitest before moving on. Visual-only changes don't skip gates.
  • Regenerate snapshots deliberately, at merge-ready time
    vitest -u updates only mismatching snapshots; passing ones untouched.
  • Run the suite again, unmodified — the critical two-pass
    Pass 1 absorbs legitimate visual drift; pass 2 surfaces what still fails. Remaining failures are real bugs, not snapshot issues.
  • Verify snapshot changes are deliberate
    Diff each updated snapshot against the phase's stated scope. A snapshot updating outside scope signals a leak.
  • Run code review on every phase
    Look specifically for high-severity semantic bugs snapshots miss — button activation logic, measurement-cache keying.
  • Respect the framework's constraints
    E.g. React 19 forbids setState in effects. Keep state transformations immutable throughout.

Gotchas
  • Never assume "all failures are probably snapshot-related" — always do the two-pass verification before concluding that.
  • Snapshot tests validate visual/markup shape, not semantic correctness — button state and cache-keying bugs pass snapshot diffs while functionally broken.
  • Keep "looks right" (snapshot) and "behaves right" (functional test / code review) as explicitly separate concerns.
  • Regenerating snapshots before confirming the visual change is intended can permanently bake a regression into the baseline.
