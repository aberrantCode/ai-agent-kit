Analyze Conversations

JSONL transcript mining · recurring-issue detection · prevention-patch proposals

Purpose · the funnel
Find issues the agent hit more than once across Claude Code sessions — recurring tool errors, rejections, operator corrections — and propose concrete changes that stop them recurring. A committed extractor reduces ~400 MB of JSONL to a few KB of structured signal before anything touches context.
Mechanical clustering
The script clusters what normalises mechanically — error signatures. Trust these rows as-is.
Judgment clustering
You cluster what needs judgment — free-text operator corrections the script only extracts as candidates.
Recurrence axis
Distinct sessions, not raw count. 50 hits in one session is one incident; the same error in 8 sessions is systemic.
five-step pipeline, read-only end to end
Pipeline · Steps 1–5
  • STEP 1
    Run the extractor
    analyze-conversations.py --window 7d — read-only, prints a bounded Markdown report to stdout. --window accepts a calendar window (7d), a session count (20s), a date, or all. Also: --max-files, --min-recurrence, --max-corrections, --include-sidechains, --project-dir.
  • STEP 2
    Read the report — three sections
    Recurring tool errors (pre-clustered, ranked by sessions then count) · operator rejections/interrupts (parallel-cancellation noise excluded) · operator corrections (bounded excerpts for you to cluster).
  • STEP 3
    Cluster and define "recurring"
    An issue counts when an error signature spans ≥2 sessions, OR ≥2 correction excerpts (different sessions) re-steer the same behaviour. Merge mechanical errors with corrections describing the same root cause.
  • STEP 4
    Map each issue to a prevention surface
    Pick the narrowest surface that actually prevents recurrence — see mapping below. Prefer a hook/check over prose when mechanically enforceable.
  • STEP 5
    Deliver list, then draft patches
    Two-stage output: a concise ranked list first, then drafted patches for approval. Do not write patches until the operator approves.
extract→ read→ cluster→ map→ list + patch
When to Use / Not Use
Use for
"What mistakes do I keep making?" · finding problems hit >1 time · periodic self-review before a sprint or after a frustrating stretch.
Not for
Debugging one live failure (use diagnostics-first + the probe table) or reviewing a single transcript already in context.
Reading-the-Data Gotchas
  • type:user is overloaded — filters out harness injections (skill bodies, system-reminders, compaction) or they drown real corrections ~80:1.
  • Trust is_error, not content-sniffing — a verifier printing "errors: 0" is not a failure.
  • Cancelled: parallel tool call is downstream noise — counted separately, excluded from ranking.
  • Long baton prompts ≠ corrections — only a lead-in re-steer counts for long turns.
Step 4 Detail · Issue Shape → Prevention Surface
Technical failure mode
Tool/command/API breaks the same way → new docs/learnings.md ISSUE-NNN entry.
Mechanically detectable
File pattern, forbidden command, missing flag → a PreToolUse hook or pre-commit check.
Hard "never do X"
Repeatedly violated constraint → docs/negative-constraints.md.
Process/behaviour correction
How the agent should work → a .claude/rules/*.md rule or memory/feedback_*.md.
Recurring diagnostic
"Doesn't work on host X" again → a read-only probe script + runbook entry.
Settled decision re-litigated
Operator preference → a memory/project_*.md or feedback_*.md file.
Limits & Validated Signal
Limits
Correction clustering is manual by design. --window all + high --max-files scans slowly (streaming keeps memory flat) — narrow the window for routine reviews. Sub-agent transcripts excluded unless --include-sidechains is passed.
Validated against the live corpus
A 7-day run over 76 sessions found Edit "File has not been read yet" across 30 sessions, 'dev' is already used by worktree across ~9, an SSH key-drift error across 3, a blocked forbidden sleep, and a correction cluster of the agent claiming completion it hadn't verified.
