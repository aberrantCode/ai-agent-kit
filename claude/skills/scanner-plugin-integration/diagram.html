Scanner Plugin Integration

merging orphaned scaffolds · one interface, many providers · gate in DryRun not Run

When to use
Integrating a set of similar external-service plugins (scanners, providers, suppliers — any "one interface, many implementations" integration) into a codebase: merging a scaffold built in a separate location into the real package tree, adding a new provider that follows an established template, or fixing/decommissioning an existing one.
fix orphaned scaffold → preserve interface shape → gate architecture → add/fix provider
1 · Fixing orphaned scaffolds (Go-specific, broadly applicable)
  • identity
    Package identity is the directory, not the package clause
    Code in a separate top-level folder is a distinct package even if the clause matches — causes undefined: suppliers.X import errors.
  • fix
    Move, don't re-export
    Move files into the real tree, merge registration into the existing init.go, delete the scaffold entirely — no re-export shim.
  • generalize
    Pattern applies broadly
    Any "sidecar extraction that should have been integrated" gets the same fix: merge directories, consolidate init/registration, delete the orphan.
2 · Preserve the plugin interface pattern
Common shape
Small fixed-method interface (e.g. 4 methods) + supplier-delegation layer + mock implementation + table-driven tests, applied consistently across every provider.
Why it matters
Keeps future providers a templating exercise, not a bespoke one-off each time.
3 · Gating architecture
DryRun() not Run()
Library.Scan calls DryRun() for every registered scanner first and skips failures — gate lives at the aggregate level, Run() stays thin.
/dryrun vs /run
HTTP /run deliberately bypasses the gate, assuming the caller already checked /dryrun — no redundant validation inside /run.
4 · Adding or fixing a provider
  • understand
    Read the supplier interface first
    Method signatures (e.g. SearchByPhone) and expected response shape, before writing code.
  • verify live
    Test the live endpoint before coding
    Confirm exact URL, auth mechanism, response format — vendor docs drift from the actual API.
  • mock
    Keep interfaces mock-friendly
    Tests run without live credentials, reinforcing the 4-method interface + mock pattern.
  • dedup
    Reuse response structs across API versions
    Migrating v1 → v2 reuses DehashedEntry-style structs instead of parallel v2 types.
  • decommission
    Removal is a checklist, not a deletion
    Backend files + role, registration, UI branches, docs, .env.example, full test suite.
Gotchas
  • Matching package clause + different directory is still a different package — produces "undefined" errors that look like a missing import.
  • Don't put gating in Run() "for safety" — duplicates the aggregate check and creates two sources of truth.
  • Vendor/API docs go stale — confirm URL/auth/response shape against the live endpoint before writing integration code.
  • Decommissioning is easy to do partially — treat it as a checklist (backend, UI, docs, env, tests), not a single deletion.
