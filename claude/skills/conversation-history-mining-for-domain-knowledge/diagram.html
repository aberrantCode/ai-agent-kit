Conversation History Mining for Domain Knowledge

mine prior Claude transcripts for load-bearing domain knowledge · not logs, not current code alone

When to Use
Load-bearing domain knowledge about a service, system, or process has been discussed before but isn't written down anywhere authoritative: building a new skill for an internal tool, onboarding onto a legacy system, or auditing "what keeps going wrong" across a fleet of repos. Transcripts capture failure modes, workarounds, and decisions that current code and logs alone won't reveal — they encode the why, not just the what.
five-step method
Method
  • 1
    Mine transcripts, not just logs or current state
    Prior conversations accumulate incidents, constraints, and workarounds that never made it into docs — e.g. a Honcho skill built by pulling DERIVER_FLUSH_ENABLED behavior, a conclusion-pollution incident, and a 6-day silent auth-key failure straight out of history.
  • 2
    Dispatch a search agent over the transcript corpus
    Don't read everything serially. Search for symptom descriptions, workaround phrasing, and decision language ("we changed X because Y", "the actual cause was").
  • 3
    Parse transcripts as structured data, not prose
    JSONL session logs carry tool calls and results as first-class fields. Use a read-only funnel script to extract them, then cluster errors by a normalized signature (strip timestamps/IDs, group by error shape).
  • 4
    Treat the transcript as the arbiter of what happened
    Current git/system state gets masked by later manual remediation. The transcript is the only record of what the agent actually did and claimed at the time; session-end wrap-ups state the intended outcome even if not fully realized.
  • 5
    Convert discoveries into durable memory: capture-then-curate
    Agents append diagnostic insights to a lightweight intake file, no ceremony. A curator agent periodically sweeps it: dedups entries, allocates stable IDs, links related items, writes into a permanent index — keeping CLAUDE.md from ballooning.
Inputs & Outputs
Inputs
Prior Claude conversation transcripts (JSONL session logs) · a dispatched search agent · a lightweight append-only intake file
Outputs
Clustered recurring failure classes · a new skill/doc/knowledge base entry · a curated, deduped permanent index with stable IDs
Gotchas
  • Don't treat current repo/system state as ground truth for "did this bug get fixed" — silent manual remediation outside the session makes a real recurring bug look resolved. Cross-check against the transcript.
  • Clustering needs normalization (strip volatile timestamps/paths/IDs) or every error looks unique and no pattern emerges.
  • An intake file that's never curated just becomes a second, messier CLAUDE.md — the curator pass (dedup + ID allocation + linking) is what makes it durable memory rather than a growing pile of notes.
  • Measuring "session completion vs. actual delivery" requires reading both the wrap-up (stated intent) and the tool-call trace (actual actions) — either alone overstates or understates what was really delivered.
