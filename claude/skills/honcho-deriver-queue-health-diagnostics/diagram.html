Honcho Deriver Queue Health Diagnostics

MCP layer vs. server-side queue · broken vs. contended · conclusion pollution audit

When to use
A Honcho instance (or structurally similar background-processing queue — ffmpeg extraction, refresh jobs, any worker draining a Postgres-backed queue) appears stalled or reports a suspicious pending/error count. Use before assuming "the pipeline is broken" — most stalls are contention, not failure.
nine-step diagnostic method
Method
  • 1
    Check both layers
    MCP layer (get_representation, get_context) only shows derivation output. Real health = server-side Postgres queue, SSH-only.
  • 2
    A healthy-looking queue can hide history
    pending=0 doesn't mean fine — query SELECT processed, error, created_at FROM queue WHERE error > 0, group by cause.
  • 3
    Check the deriver process itself
    For active/ongoing errors: is it running, CPU/memory, live error logs.
  • 4
    Reset historical errors safely
    Only after acquiring the coordination lock: UPDATE queue SET error=0 WHERE error>0 AND error NOT IN ('reconciler','dream').
  • 5
    Static pending count → diagnose in order
    See branch below — deriver running? preference vectors configured? scheduler enqueuing?
  • 6
    Check inProgressWorkUnits
    Should be >0 while actively draining. Zero alongside nonzero pending confirms the deriver isn't picking up work.
  • 7
    Verify live state, not memory
    Stale assumptions ("we run it locally") are common — check current container/process state and origin/dev config.
  • 8
    Shared-resource pipelines: falsifiable criteria
    Define "broken" up front (see three-check panel below) before concluding contention vs. defect.
  • 9
    Audit pollution in stored conclusions
    Enumerate every peer-scope, count raw conclusions, inspect samples for hallucination/truncation/misattribution, trace to origin.
Decision tree · "pending count is static"
a. Is any deriver process actually running?
Check container logs, systemd status.
b. Do pending items have users with preference vectors configured?
Check config.
c. Are analyses actually being enqueued?
Trace the scheduler path.
A static count with zero active deriver is the single most common root cause — check (a) first.
Falsifiable "broken" criteria · shared-resource pipelines
Task state ≠ failed
Explicit failure state check.
Recent log activity
Extraction timestamps within expected quiet window.
Disk/permissions OK
Rule out resource-exhaustion false positives.
Conclude "broken" only if (a) or (b) actually fail. If all three pass but throughput is low, it's contention (e.g. a competing transcode job), not a defect. Build one diagnostic script that checks all signals in a single pass.
Gotchas
  • Don't stop at the MCP layer — the actual queue state lives in Postgres, SSH access required.
  • A flat pending count is ambiguous — "broken" (no deriver) vs. "healthy but blocked upstream" (no preference vectors, scheduler not enqueuing). Check all three causes.
  • Never bulk-reset queue errors without excluding reconciler/dream and without the coordination lock — a concurrent reset can race an active deriver.
  • Resource contention looks identical to "broken" from throughput alone — always run the three falsifiable checks first.
  • Pollution cleanup should be targeted (trace to origin), not wholesale deletion.
