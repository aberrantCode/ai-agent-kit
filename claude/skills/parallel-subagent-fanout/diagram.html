Parallel Subagent Fanout

dispatch by lens, not by chunk · reconcile into one verdict

When to use
Use when a task splits into independent lenses or disjoint-file subtasks — multi-layer system audits, large documentation backfills (10+ specs), batch state-transition decisions, or code/security review of a feature branch — to dispatch multiple subagents concurrently and reconcile their results into one verdict.
Multi-layer audit
Each layer gets distinct domain expertise — node inventory, topology, extension patterns.
Doc backfill
10+ specs grounded independently against source code, in parallel.
Branch review
code-reviewer + security-reviewer run concurrently against the same diff.
ten-step method, one dispatch cycle
Method · partition → dispatch → reconcile
  • 1 · Partition
    By lens, not by chunk
    One node type, one doc topic, one review discipline, one audit axis per subagent — produces fresh specialized findings instead of N shallow copies.
  • 2 · Re-baseline
    Sync before fanning out
    Pull latest origin/dev (or equivalent) immediately before dispatch. Stale local state is the #1 cause of duplicated work.
  • 3 · Brief
    Everything needed, nothing unusable
    Subagents run blind to each other and can't see images — translate screenshots/diagrams into written summaries first.
  • 4 · Scope diffs
    Name the diff target explicitly
    e.g. origin/dev...HEAD, so reviewers evaluate only the delta. Concurrent code+security review is mandatory on admin/auth/mutation surfaces.
  • 5 · Respect deps
    Fanout only for disjoint files
    Shared-file or shared-role subtasks get sequenced (PR-A merges before PR-B starts), never parallelized.
  • 6 · Cross-validate
    Collect, don't just concatenate
    Verify every claimed issue against the real source of truth. Drop claims that can't be traced to a concrete value or line.
  • 7 · Reconcile
    Disagreements → root cause
    Trace conflicting findings back to why (e.g. a timing issue on an unmerged branch) rather than treating both as equally valid.
  • 8 · Consensus
    Threshold for batch decisions
    Auto-apply verdicts at ~80%+ subagent agreement; surface only the ambiguous minority for user judgment.
  • 9 · Synthesize
    Owned by the main session
    Merging findings, visual judgment calls, and the final unified report stay with the dispatcher — never delegated to a subagent.
  • 10 · Verify
    Cross-artifact consistency pass
    For doc backfills or multi-file writes, check frontmatter uniformity and cross-file consistency once all subagents finish.

Disjoint files → parallelize
Fan out
Independent lenses touching separate files or roles — different node types, different audit axes, code-reviewer vs security-reviewer on the same diff. Dispatch concurrently, reconcile once.
Shared files/roles → sequence
Serialize instead
Two subtasks touching the same file or role: let PR-A merge before PR-B starts. Parallel edits to shared surfaces cause conflicts, not speedups.
Inputs → Outputs
In
A splittable task (lenses or disjoint-file subtasks), a re-baselined repo state, and per-subagent briefs — including written translations of any screenshots/diagrams.
Out
One reconciled, cross-validated verdict or artifact set — traceable findings only, disagreements resolved by root cause, consistency-checked across files.
Gotchas
  • Skipping the re-baseline step is the most common failure — stale state makes subagents "discover" and redo work that already landed.
  • A subagent reporting "this work already exists" is a valid, valuable result — it surfaces deduplication problems the dispatcher would otherwise miss.
  • Never hand a subagent raw claims to trust blindly; unverified findings that don't trace to a concrete source value get dropped, not merged in.
  • Fanning out edits to overlapping files/roles causes merge conflicts — check the dependency graph before assuming everything is parallelizable.
  • Subagents can't see screenshots or other images; skipping a written translation of visual context silently degrades their output.
