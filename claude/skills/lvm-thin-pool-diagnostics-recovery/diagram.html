LVM Thin-Pool Diagnostics & Recovery

layered diagnosis · pool → filesystem → space reclaim · never trust df alone

When to use
ENOSPC despite free space
A container or VM reports free filesystem space (df) but writes fail with ENOSPC, or the filesystem has gone read-only.
Host-level out-of-data-space
Symptoms suggest an LVM thin pool is out of data space (error IO mode).
Unpredictable write stalls
Streaming/write operations to thin-backed storage stall under load even when the pool isn't reported full.
diagnose in layers — never trust filesystem-level free space alone
Layer 1 · Three-layer diagnosis
Container / guest level
df can show "free" space that's meaningless if the backing thin pool can't allocate blocks — every write fails with ENOSPC regardless.
Host level
Run lvs for true thin-pool utilization; dmesg for out-of-data-space error IO mode — pool refuses all new allocations fleet-wide, not just one guest.
Filesystem level
Check for a stuck flag — ext4's errors=remount-ro can leave it in emergency_ro even after the underlying pool issue is fixed.
Layer 2 · Recovery order — pool, then filesystem, then reclaim
lvextend -L+<sizeG>→ check mount | grep rw→ e2fsck (if still ro)→ restart guest→ fstrim from HOST
  • Extend
    Extend the pool live, no downtime
    lvextend -L+<sizeG> pve/data — the pool exits error-IO mode once resumed.
  • Unstick
    Filesystem stays stuck read-only
    Check mount | grep rw; if still ro, a bare remount is rejected because the ext4 superblock error flag persists.
  • Clear flag
    Clear the superblock error flag
    Stop the guest, run e2fsck on the volume, then restart it.
  • Reclaim
    Reclaim freed blocks — from the HOST
    pct fstrim <ctid> (Proxmox). Unprivileged fstrim inside the guest fails; the privileged host-side call succeeds, enabling clean reinstall of corrupted packages.
Layer 3 · Write stalls ≠ full-pool ENOSPC — diagnose separately
Thin-pool writes can silently stall under metadata pressure (not just data-space pressure) at predictable thresholds — blocking streaming operations without an outright error.
Confirm
Use iostat to check I/Os-in-progress, sample /proc/diskstats over time, correlate against live-mirror/replication activity.
Recover
Drain the near-full pool onto a healthy one using idempotent offline moves — never online moves under pressure. Force-stop the guest/service first to prevent new writes cascading into the source pool mid-drain.
Gotchas
  • Don't stop at "container shows free space" — the classic false signal; always check lvs/dmesg on the host first.
  • A plain remount,rw after fixing the pool will not clear an emergency_ro ext4 volume — e2fsck is required first.
  • Don't run fstrim unprivileged from inside the guest expecting reclamation — it silently fails or no-ops; must run privileged from the host.
  • Pool exhaustion is fleet-wide — recovering one guest doesn't mean the pool is safe; verify lvs shows healthy headroom before closing the incident.
  • When stalls (not hard failures) are the symptom, don't assume "fine" just because it isn't 100% full — metadata pressure stalls writes below that threshold; always drain offline once confirmed.
