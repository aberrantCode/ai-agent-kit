Project Plan / Task Reconciliation

worker task → plan/backlog reconciliation · claim-verification · pipeline-bypass audit

Purpose · when to use
Reconciles a completed worker task against the project plan and backlog — appending a parseable completion block, updating plan status and archives, verifying subagent claims against actual git/repo state, and auditing for features that shipped without a formal plan/task trail.
After a worker finishes
Run reconciliation as soon as a subagent completes a task, before trusting its report.
Before advancing
Gate the next task on a clean reconciliation of the one that just finished.
Auditing overall status
Periodic sweep across the whole plan/backlog, not just the latest task.
the reconciliation loop
Method · core loop
  • 1
    Append a Completion block
    Parseable contract on the task file: Status · Summary · Artifacts · Tests · Notes. Don't skip fields even if "obvious."
  • 2
    Verify the claims
    Cross-check every claim ("already merged", "endpoint returns 200") against real git status / git log / a live curl before propagating it forward.
  • 3
    Branch on Status
    success / failure / blocked drive three different remediation paths — see decision panel below.
  • 6
    Refresh the what's-next doc
    Every reconciliation updates FOCUS.md (or equivalent) with next action + current blockers so a fresh session can resume cold.
  • 7
    Respect the dependency graph
    Sequence work by each feature's depends_on; never start a downstream feature's tasks before its dependencies show done.
  • 10·11
    Track discoveries + audit bypass
    Durable cross-cutting facts go to a project-wide index doc; periodically cross-reference the feature list against the task-count matrix for pipeline-bypass.
append→ verify→ branch→ refresh next-action→ respect deps→ audit bypass
Decision · branch on Status field
success
Mark the plan task done, move the task file to the archive, update plan frontmatter (failures count, last_updated).
failure
Diagnose root cause, increment the plan's failure count, insert corrective tasks before the failed one, re-run from the corrective task — not from scratch.
blocked
Do not archive. Surface the blocker in FOCUS.md so the next session picks it up first instead of silently skipping it.
Rule · bookkeeping PRs
Plan/backlog changes get their own PR
Never reconcile plan bookkeeping direct-to-integration-branch — this forces the same review discipline on backlog edits as on code.
Rule · frozen target file
Freeze protocol → split into two PRs
One PR for the substantive change (migration, lint rule, SoT mirror), one purely for the backlog-closure edit — never bundle them or you race the freeze.
Implemented, no plan
Process debt — not a delivery gap
Feature shipped with zero recorded tasks: it bypassed the formal pipeline. Flag for retroactive documentation or a prospective enforcement fix.
Unimplemented, has a plan
Real remaining work
This looks similar to process debt in a status report but demands a completely different response — conflating the two misdirects remediation effort.
Inputs & Outputs
Consumes
Task file's completion claims & subagent hand-off prompt · live git/repo state (status/log/curl) · plan frontmatter + depends_on graph · full feature/backlog list.
Produces
Parseable ## Completion block · updated plan status & archived task file · refreshed FOCUS.md next-action · corrective tasks inserted · plan/bookkeeping PR · pipeline-bypass flags.
Gotchas
  • A subagent's self-reported hand-off prompt is a claim, not a fact — always cross-check against git/repo state before re-emitting it.
  • Architectural drift can creep into hand-off prompts too — catch and downgrade it during reconciliation, not after the next session has already acted on it.
  • "Implemented but missing a plan" and "unimplemented with a plan" look similar in a status report but require opposite responses.
  • Freeze protocols on backlog directories are easy to violate by bundling the closure edit into the substance PR — always check frozen status before deciding PR boundaries.
