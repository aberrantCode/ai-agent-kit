Honcho

AI-native memory backend for stateful agents · peers → sessions → deriver → representation

Mental model
Peers
Any participant — human or AI, modeled the same way. observe_me / observe_others control what gets reasoned about.
Sessions
Scope conversations between peers.
Messages
Raw data fed in. No messages → no reasoning → no memory.
Deriver
Background reasoning model that extracts premises, draws conclusions, builds each peer's representation asynchronously.
Query memory via peer.chat(query) (reasoned answer) and/or session.context() (formatted history + representations).

Quickstart flow
install SDK→ honcho.peer()→ honcho.session()→ session.add_messages()→ peer.chat(query)
honcho = Honcho(workspace_id="app", api_key=...)
user = honcho.peer("user-123")
assistant = honcho.peer("assistant", configuration=PeerConfig(observe_me=False))
session = honcho.session("conversation-1")
session.add_messages([user.message(...), assistant.message(...)])
print(user.chat("What does this user prefer?"))
Python (sync)
uv add honcho-ai
TypeScript (async by default)
bun add @honcho-ai/sdk
Rules of thumb
  • One workspace per application — don't spread data across workspaces.
  • Create peers for AI assistants too, with observe_me=False — otherwise the AI's own narration pollutes the user's representation.
  • Always store messages after each exchange — that's what feeds the deriver.
  • Never block on the deriver — derivation is asynchronous; don't poll for the queue to be "empty."
  • When debugging memory, verify both surfaces — what clients read back, and the server-side deriver queue. A healthy API can sit on a deriver that's failed for days.
