Deployment Driver: Pin Rewrite from Release-Tag Source of Truth

Use when a deployment/build system has both a human-editable "intent" field (release tag, plan row) and a derived "pin" artifact (image pin file, lockfile) — edits always go to the source of truth; the driver re-derives the pin.

Core Contract · edit the source, not the shadow
A privately-built image has two things that both look like "the version": a release tag checked into the repo, and a pin file generated at build time. Only one of them is authoritative — hand-editing the other is a silent no-op.
Source of Truth — release tag
Lives in the build script, checked into the repo, human-editable. This is where an intended version change belongs.
Derived Artifact — pin file
Generated at build time as tag:sha. A cache of the driver's last derivation — never a place to make decisions.
edit release tag → driver rebuilds → pin file auto-rewritten
Method · five-step discipline
  • 1
    Identify source of truth vs. derived artifact
    The release tag is authoritative; the pin file is derived at build time (release tag + commit SHA baked in).
  • 2
    Always edit the source-of-truth field
    Never hand-edit the derived pin — the next build silently re-derives it from the (unchanged) release tag and re-deploys the old version.
  • 3
    Let the driver own the derivation
    The deployment driver reads the release tag, builds with the commit SHA appended, and auto-rewrites the pin file as a build side-effect.
  • 4
    Apply the same discipline to orchestration loops
    A /loop-style driver re-reads its source of truth fresh every iteration — git fetch + git log for the current dev tip, then re-read plan rows. Never trust chat history or "what I did last iteration."
  • 5
    Define explicit pause conditions
    So the loop doesn't spin or act on stale/ambiguous state.
When to Use
Build-time pin, repo-checked-in tag
A privately-built image has a version pin file generated at build time, separate from a release tag checked into the repo.
Designing/debugging a deploy loop
Needs to decide "what's the next version/task to act on" without trusting stale chat history or memory.
"The deploy did nothing" report
Someone reports a deploy did nothing after what looked like a version bump — classic symptom of an edited pin, unchanged tag.
Loop Pause Conditions
  • No eligible todo task found in the plan source.
  • A release step isn't authorized yet.
  • Token/usage budget is approaching its limit.
Gotchas
Silent no-op
Hand-editing the pin file produces no error — the system quietly redeploys the old version on the next build. Easy to miss in review.
Two things claim to be "the pin"
A checked-in tag and a derived file both look authoritative — verify which one the driver actually reads at build time before assuming an edit takes effect.
Loops that reason from their own output drift
A loop that trusts its own prior output instead of live repo state drifts — always re-fetch and re-read the plan/tag source per iteration rather than carrying forward an in-memory "current version."
