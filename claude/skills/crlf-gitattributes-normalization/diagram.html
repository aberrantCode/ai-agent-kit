CRLF / .gitattributes Normalization

fix spurious line-ending noise at the repo level · never reformat the file or skip the test

When to Use · trigger symptoms
In a Windows/Linux-mixed repo, use this whenever a diff, lint, or test failure looks like a content problem but is actually a line-ending artifact — normalize via .gitattributes rather than reformatting the file or skipping the test.
Spurious linter warnings
prettier / eslint / gofmt report "Delete ␍" or "file not formatted" even though the file looks clean.
Golden-fixture test failures
A byte-exact comparison against a fixture fails only on Windows, or only after a fresh checkout.
About to mask the symptom
You're about to re-run a formatter or add a test skip/guard — stop and check whether it's a line-ending artifact first.
five-step diagnose → fix → verify method
Method
  • 1
    Understand the mechanism
    Git blobs are always LF. With core.autocrlf=true, git rewrites LF → CRLF in the working tree on checkout — linters see the CRLF copy even though the committed blob is clean.
  • 2
    Diagnose before "fixing"
    Confirm the committed blob is clean: git show <file> | file -. If it reports LF with no CRLF mention, the warning is harmless working-tree noise.
  • 3
    Fix at the repo level
    Add .gitattributes rules — never a per-developer git config. General: * text=auto eol=lf. Fixtures: **/testdata/**/*.txt text eol=lf.
  • 4
    Re-normalize the working tree
    General pass: git checkout HEAD -- .. Targeted fixtures: git add --renormalize .. Then re-run the linter/formatter/test.
  • 5
    Confirm idempotency
    Blobs were already LF — only the working tree changes. Re-running checkout/renormalize on an already-normalized tree must be a safe no-op; verify before calling it done.
Decision Point · is the blob clean?
git show <file> | file - — does the committed blob mention CRLF?
No CRLF mention → blob is LF
Harmless working-tree artifact. Fix via .gitattributes + re-normalize. Do not reformat the file or add a test skip guard.
Blob itself contains CRLF
Not a working-tree artifact — this method doesn't apply; the content genuinely needs fixing.
Gotchas
  • Don't reformat a file or skip/guard a test for a CRLF-vs-LF mismatch — check git show | file - first.
  • * text=auto eol=lf won't retroactively fix already-checked-out files — still re-checkout or renormalize.
  • Fixture/golden-file tests need their own explicit eol=lf rule scoped to the fixture path.
  • Avoid git config core.autocrlf=false per-machine — it only fixes it for one contributor.
Inputs → Outputs
Input · mixed-checkout repo
Windows/Linux contributors with varying core.autocrlf, plus a spurious lint/format/byte-exact-fixture failure.
Output · repo-level rule + clean tree
A committed .gitattributes rule (general and/or fixture-scoped), a re-normalized working tree, and a passing, idempotent lint/test run.
