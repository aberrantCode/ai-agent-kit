Accumulated Feature Branch Workflow

shared feature branches · risk-seam PR splits · mixed dirty/committed shipping

When to use
Use when implementing multiple related enhancements, or a large feature that must be split across risk boundaries, and you need to decide branch/PR structure — accumulating related work on one shared branch with atomic commits per enhancement, splitting risky multi-PR features along risk seams, and safely handling a branch that has both committed and dirty (uncommitted) work when shipping.
Bundle of enhancements
Several related, independent improvements that belong together conceptually.
Large feature, mixed risk
One feature spanning parts with different blast radii — CPU-only vs. shared-auth vs. GPU/infra.
Mixed branch state
Shipping a branch that has both pre-committed work and uncommitted dirty files.
six method steps, one shared branch
Method
  • 1
    Accumulate on one shared branch
    e.g. feat/optimizer-enhancements-bundle. One atomic conventional commit per enhancement; open a single combined PR only once all enhancements pass tests. Preserves per-enhancement git blame/bisect history.
  • 2
    Split large features along risk seams
    Not arbitrary size. Each PR gets its own verification gate; never merge without live-verify evidence — call out any missing gate explicitly rather than skipping it silently.
  • 3
    Cross-session orchestration
    iterate-tasks style: advance exactly one PR per invocation; let the next-session prompt carry context forward instead of bloating the parent session.
  • 4
    Strict PR-only discipline
    Rebase on merge to avoid phantom-endpoint artifacts. Own contract doc + feature branch per sub-task, backend-first ordering when frontend consumes new endpoints. Archive completed task files with closure narratives.
  • 5
    Detect mixed branch state early (Step 0)
    git status + git rev-list to separately count commits-ahead-of-dev vs. dirty files. If both exist, ask the user — never guess.
  • 6
    Verify relatedness before combining
    Diff committed set against dirty set. Never blind git add --all; flag throwaway files for exclusion; stash unrelated pre-existing WIP separately so it survives untouched.
Decision Point · mixed branch state at ship time
(a) Ship committed only
Leave the dirty files behind untouched; ship exactly what's already committed on the branch.
(b) Fold dirty into a new commit
Only after confirming the dirty set is genuinely related to the committed set — then ship both together.
Ambiguity between (a) and (b) is never resolved by guessing — always ask the user explicitly.
Risk-Seam Split · example lifecycle
PR-1 · reversible
CPU-only, fully reversible change. Still requires its own live-verify gate — "safe" doesn't mean exempt.
PR-2 · shared substrate
Touches shared ingress/auth. Higher blast radius; rebase before merging to avoid phantom endpoints.
PR-3 · infra risk
GPU/infra-risky change, e.g. two consecutive Ansible runs reporting changed=0 as the gate.
PR-1 verified→ rebase→ PR-2 verified→ rebase→ PR-3 verified
Gotchas
  • A "bundle" branch only works when the enhancements are genuinely independent and reviewable together — don't use it to hide an oversized, poorly-scoped feature.
  • Skipping the live-verify gate on any PR in a risk-seam split (even the "safe" reversible one) breaks the whole point of the split — the safety guarantee is per-PR, not just at the end.
  • git add --all on a mixed dirty/committed branch is the most common way unrelated scratch files (mockups, local notes) end up shipped — always inspect dirty files individually first.
  • Rebase-on-merge for multi-PR features prevents phantom endpoints, but only if every dependent PR actually rebases after each merge — a stale branch will silently reference deleted/renamed code.
