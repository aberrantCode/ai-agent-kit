Brand Token Extraction and Documentation

real palette from raw CSS · recreated assets · versioned named tokens

When to Use
Reskinning an app with a real brand's visual identity: extract the actual palette from a live site's raw CSS (never markdown), recreate logo/icon assets programmatically with documented extraction rationale, and codify everything as versioned, named design tokens rather than one-off hardcoded values.
Live site, no source access
Pull the real palette straight out of the stylesheets the browser actually loads.
Client logo/asset pack
Inspect the rendered mark and rebuild its construction programmatically.
Existing design system
Derive new tokens from established primitives instead of a parallel scale.
phase 1 · extract from a live site
Extracting · steps 1–5
  • 1
    Fetch raw HTML/CSS, never markdown
    Markdown conversion strips all styling — exactly the data needed.
  • 2
    Pull every linked stylesheet
    curl the page, grep <link> tags, fetch each linked CSS file.
  • 3
    Rank colors by occurrence count
    Find the real palette across the combined stylesheet — don't take the first hex values seen.
  • 4
    Skip page-builder default colors
    e.g. Divi's #2ea3f2 — frequent, but an artifact of the framework, not a brand choice.
  • 5
    Recreate imagery programmatically
    No official source file? Inspect the rendered SVG/PNG construction (two-tone split, italic wordmark) and rebuild it — don't approximate by eye.
phase 2 · capture & document as tokens
Capturing & Documenting · steps 6–10
  • 6
    Record both color spaces
    OKLCH for perceptual editing/theming, sRGB hex for contexts needing a literal value (data-URIs).
  • 7
    Comment source + flag "baked" values
    Note the source CSS file per token; flag values hard-coded into a generated output — they need manual re-sync when the source token changes.
  • 8
    Light + dark theme variants
    All four-value substitutions for a complete palette swap — not just the primary color.
  • 9
    Version-control the real values
    Never leave a generic/placeholder token doc in the repo once real brand values are known.
  • 10
    Token names are the API
    Rename/remove = MAJOR (breaks every consumer). Add a new token = MINOR.
Extending an Existing System · steps 11–12
Derive, don't invent
New tokens come from existing primitives — e.g. a "projection" type scale derived from container-query percentages of the base ramp, so it inherits proportional relationships instead of drifting.
Reproducible asset generation
Document extraction rationale in the PR, commit a high-res master asset, build a script that deterministically scales/keys/crops from it — re-provisioning becomes a script re-run.
Gotchas
  • Markdown-converted content silently discards the CSS you need — always fetch raw HTML/CSS.
  • Page-builder frameworks (Divi and similar) pollute occurrence-count ranking with signature default colors.
  • A baked token is a hidden maintenance liability — comment it or it silently drifts from the source.
  • Token names are the stable contract, not just values — a name change/removal breaks downstream.
End-to-End Lifecycle
fetch raw CSS→ rank colors→ filter defaults→ recreate assets→ record OKLCH+hex→ flag baked→ version tokens→ extend system
