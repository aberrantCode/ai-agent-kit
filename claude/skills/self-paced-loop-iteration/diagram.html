Self-Paced Loop Iteration

/loop with no fixed interval · one shippable unit per wakeup · pace set by the work, not the clock

When to use
Draining a multi-task backlog, feature plan, or long-running operational workload — a phased feature plan, a backlog of 5-50 small tasks, an operational batch job, or open-ended exploration. Use /loop with no fixed interval so duration is determined by the work, not the clock.
each iteration: ground truth → one unit → gate to green → sync doc → next wakeup
Method — per iteration
  • 1
    Re-establish ground truth
    Re-read the progress doc, confirm git state via fetch + reflog — never trust a prior turn's summary.
  • 2
    Scope to exactly one shippable unit
    One task, one enhancement, one phase item — never bundle multiple backlog rows.
  • 3
    TDD for code changes
    RED → GREEN → refactor, then full gate suite (ruff, mypy --strict, pytest / tsc, lint, vitest) green before commit.
  • 4
    Code review inside the iteration
    Not deferred to the end. Address CRITICAL/HIGH immediately; security-review pass for auth/admin/backup touches.
  • 5
    Size discipline
    Warn at 400 changed LOC, hard-cap at 800 — split the unit rather than ship oversized.
  • 6
    Sync the progress doc in the same commit/PR
    Checkmark + YYYY-MM-DD + one-line resolution note — doc drift is the top way loop work becomes untrustworthy.
  • 7
    Merge with a regular merge commit
    Preserves full per-iteration history on the integration branch.
  • 8
    Decide the next wakeup dynamically
    Emit the result into context, pick the next backlog row / phase-gated item / newly-discovered task, schedule a short wakeup.
  • 9
    Fresh-context iterations
    Dispatch each iteration as a fresh subagent with full task context but not accumulated conversation history — avoids context exhaustion.
  • 10
    Checkpoint before large or risky phases
    A self-paced loop is not a license to skip approval gates on big-ticket phases.
Resumption model
interrupted / restarted→ re-read progress doc→ re-confirm origin/<branch>→ resume — safe after any merged unit
Gotchas
  • Must tolerate multi-session resumption — never resume from stale in-context assumptions.
  • Phase gating matters: don't start a downstream task before its documented predecessor is actually complete, even if the loop "has time."
  • Self-pacing is about cadence, not about skipping verification — every iteration still needs gate-to-green and review.
