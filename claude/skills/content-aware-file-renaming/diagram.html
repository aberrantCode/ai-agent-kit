Content-Aware File Renaming

content → identity → structured filename · AskUserQuestion-only prompts · six-phase pipeline

Purpose
Rename batches of content-anonymous files — downloaded statements, invoices, tax forms, receipts, contracts, scans, zip archives — into a structured name derived from what's inside the file, not its current filename or filesystem timestamps.
Naming Formula
%who% - %what% - %where% - %when% - %extra%
%who% · %what% required
Originator + doc type/subtype. Pershing - Monthly Statement
%where% optional
Redacted account/contract # or location. xxxxx6099
%when% required · %extra% optional
Date the doc applies to, granularity per type. 2024-09
six phases, in order — no skipping ahead
Workflow · six phases
  • Phase 1
    Verify the source set
    Confirm source dir, target dir, date window, file types via AskUserQuestion. Glob + mtime filter to list matches. Extract any zips into _rename_inspect/.
  • Phase 2
    Pre-analyze a representative sample
    Inspect 3–7 files spanning the batch (pdftotext, exiftool, docx/eml/csv readers). Build a mental model of issuers, doc types, date granularity, where-signal.
  • Phase 3
    Confirm formula nuances
    Lock in issuer naming, account masking, subtype usage, date format per type, non-target/out-of-scope handling, dedup aggressiveness — all via AskUserQuestion with 2–4 concrete options.
  • Phase 4
    Classify every file
    Script (not by hand past ~10 files) extracts {issuer, doc_type, where_value, when_value} + sha256 per file into classify_manifest.json. Iterate until zero unknown.
  • Phase 5
    Build dedup-and-rename plan
    Group by (issuer, what, where, when), pick canonical source, build planned filenames, resolve collisions. Write plan.json — no destructive action yet.
  • Phase 6
    Execute: copy, verify, then delete
    copy → sha256-verify → delete, always in that order. Abort on first hash mismatch, leave both sides intact.
verify→ sample→ confirm→ classify→ plan→ execute
Decision Point · dedup logic
Group by logical identity, not byte hash
Portals re-render the same doc on every download with fresh timestamps embedded — SHA256 differs, content doesn't. Group by (issuer, %what%, %where%, %when%), everything but %extra%.
Canonical-source tie-break
Prefer structured zip entries over generic loose copies; tie-break by smallest file size.
Decision Point · destructive sequence
Never move in one step
An interrupted move leaves files in a partial, hard-to-recover state. Copy first, always.
Verify before delete
Re-hash every destination against the recorded source sha256. First mismatch aborts the whole run.
Iron Rule · non-negotiable
No prompt to the user that is not an AskUserQuestion tool call. Never print a question as plain text and wait. AskUserQuestion is deferred — call ToolSearch with query: "select:AskUserQuestion" once per session before first use. Applies to every phase: source-set confirmation, formula nuances, and final execute-plan approval.
