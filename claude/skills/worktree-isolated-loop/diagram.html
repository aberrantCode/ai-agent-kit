Worktree-Isolated Loop

isolated feature-branch worktree · bounded batches · git state, not chat memory, drives the loop

When to Use
Multi-turn or batch agent work (refactoring, feature loops) that must stay isolated from concurrent edits in the primary checkout — and a self-perpetuating /loop that hands off context between turns via git state, never chat memory.
Concurrent edits risk
Primary checkout might be touched by the operator or another session at the same time.
Batch refactors
Many similar files (e.g. a family of shell scripts), each batch reviewable/mergeable independently.
eight-step method
Method
  • Initialize the worktree once per feature branch
    scripts/agent-worktree-setup.sh <branch> → .worktrees/<branch>/. Never touch the primary checkout for in-flight work.
  • Discover work in coherent, bounded batches
    Grep candidates, filter a skip list, pick one name-glob cluster at a time. Keep each batch under ~800 LOC.
  • Migrate with judgment, not blind substitution
    Per-file edits — never a blind sed across a whole cluster.
  • Verify before committing
    Syntax check, imports precede first use, no orphaned variables, smoke test the contract.
  • Log the batch
    Prepend a progress.md entry with scope + new skip categories for later batches/sessions.
  • Commit, rebase, merge from the worktree/primary
    Commit + rebase onto origin/dev from the worktree; merge from the primary checkout to avoid Windows file-lock issues.
  • Guard against same-branch collisions
    Worktrees isolate across branches, not within a shared branch. Check MERGE_HEAD/REBASE_HEAD/CHERRY_PICK_HEAD before any commit.
  • Design the loop to trust git state, not chat history
    Each iteration: git fetch + git log, re-read plan file, derive next-action from repo state. Pause when no eligible task remains, release unauthorized, or token budget nears limit.
setup→ discover→ migrate→ verify→ log→ commit/merge→ teardown
Gotchas
  • Never merge from inside the worktree on Windows — file locks during fast-forward can conflict; always merge from primary.
  • rm -rf on teardown can fail with a transient Windows lock even after git worktree remove succeeds — retry prune + rm -rf.
  • A worktree does not protect two sessions colliding on the same branch — always check in-progress merge/rebase/cherry-pick markers first.
  • A loop reasoning from chat history instead of re-reading git/plan state each iteration will drift and duplicate or skip work.
