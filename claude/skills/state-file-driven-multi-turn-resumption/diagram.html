State-File-Driven Multi-Turn Resumption

durable progress file as single source of truth · one step per turn · copy-ready handoff

When to Use
A task spans multiple sessions, context resets, or /loop iterations, and progress must survive them. A durable state file (e.g. docs/progress.md) becomes the single source of truth — not conversation history, not assumptions from the prompt.
Too large for one window
Explicitly run via /loop or recursive re-invocation across fresh sessions.
"Continue" / "resume" prompts
You may not have full history for the repo — the state file fills the gap.
seven-step method, repeated every turn
Method · one turn, one step
  • Read the state file first
    Before planning anything. Trust it over git log alone for "what's already done."
  • Verify the premise before committing effort
    Check SoT (docs/features/, docs/plans/), git log -S, in-flight PRs, the state file, and CLAUDE.md conventions. ~5 minutes, prevents redoing shipped work.
  • Advance exactly one step per turn
    No batching. Record evidence inline: PR number, output, API excerpt, commit hash, verification-gate result.
  • Respect workflow constraints
    Feature branches only, never direct commits to dev/main. Handle leftover uncommitted state from prior breaks first.
  • Use explicit completion sentinels
    Unambiguous markers — not just prose — so a fresh agent can parse "what's next" mechanically.
  • End with a copy-ready resume prompt
    Done/next, verification discipline, recurring failure-mode warnings, exact workflow to repeat — a self-perpetuating template.
  • Pause proactively for /loop
    Around ~200k tokens of context, emit "RESUME PROMPT" as the literal last output so a fresh session can paste it with zero state loss.
Gotchas
  • Acting on a prompt's stated premise without checking SoT/git log/progress file causes redundant work on already-shipped features.
  • Trusting git log alone instead of the state file (or vice versa) — reconcile both; git is the tie-breaker for "did this actually land."
  • Batching multiple checklist steps per turn defeats the auditability purpose.
  • Committing directly to dev/main instead of a feature branch breaks workflow compliance.
  • Letting context run past ~200k tokens without emitting a resume prompt loses unwritten state.
  • Omitting evidence turns checkboxes into unverifiable claims.
  • Forgetting to surface recurring failure modes lets the same mistake repeat across sessions.
  • Ignoring leftover uncommitted state from a prior session break can silently clobber or duplicate work.
