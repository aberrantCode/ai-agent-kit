Iterative Audit Gate With Streak Reset

two consecutive clean rounds required · any finding resets the streak to zero

When to use
Deliverable review
Design, feature, presentation — against a spec plus an evidence base, using independent auditors.
Verification gates
Tests, lint, type-check enforced after each commit before something ships.
Backlog / KB closure
Where a structural invariant (e.g. an index-to-detail bijection) must hold.

A single clean pass is not sufficient evidence of correctness — fresh eyes on a subsequent round routinely surface issues a prior pass missed. Any finding, however minor, resets the streak to 0 — never resume a partial count.
Method
  • 1
    Track a streak counter
    Gate = continue until two consecutive rounds both come back with zero findings.
  • 2
    Reset on any finding
    Fix it, then re-run the entire gate from round 1 — this forces genuine convergence.
  • 3
    Use independent auditors per round
    e.g. design auditor + technical auditor run separately against the same spec + evidence, so each round has a real chance of catching what others missed.
  • 4
    Document resolved decisions in the brief
    Not just in chat — lets auditors ratify a deliberate compromise instead of re-flagging it every round.
  • 5
    Structural-invariant gates
    Treat constraints as design contracts. Removals enforced strictly, additions freely — restore an orphan's missing half before removing symmetrically. Use --strict checker mode; silence = success, not a hang.
  • 6
    Gate docs/KB additions the same way
    Bijection checks, cross-reference validation, staleness warnings — treat any failure as a data-integrity issue, not an optional warning.
  • 7
    Code-quality gates need two consecutive passes
    Diagnose test-isolation bug vs. real defect, fix the actual cause atomically, restart streak at 0. The gate rerunning clean is the system of record — not "looks fine to me."
Gotchas
  • Don't resume a partially-built streak after a fix — restart from 0. A "targeted" fix can have side effects the next full pass would catch.
  • A gate that emits nothing on success can look identical to a hung or misconfigured run — verify the script's actual success contract before trusting silence.
  • Recurring stale-task classes ("fixed in a follow-up PR but never closed") indicate a verify-before-promote discipline gap — treat repeats as a process signal.
  • Independent auditors lose value if given the same framing/bias each round — vary the review lens, or the second "independent" pass just re-confirms the first.
