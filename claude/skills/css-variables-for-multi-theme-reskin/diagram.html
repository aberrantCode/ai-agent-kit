CSS Variables for Multi-Theme Reskin

token substitution, not a rebuild · light/dark + multi-brand · structure and interaction stay untouched

When to Use
Any interface that needs more than one visual identity — light/dark mode, multiple brand palettes, or a "create a themed copy of this existing dashboard" request. Also applies when explicitly asked to restyle, retheme, or reskin something that already works: the goal is a pure token substitution, not a rebuild.
In scope
Swapping --brand, --surface, --accent values per theme/brand while HTML and JS stay identical.
Out of scope
Restructuring markup or changing interaction/behavior "to make the retheme work" — that means the pattern was left.
six-step token method
Method · one palette, N themes
  • 1 · Define
    Base tokens as CSS custom properties at :root
    Name semantically — --brand, --muted, --surface, --accent — not by literal color, so the same name means "the right color for this theme" everywhere.
  • 2 · Scope
    Theme variants via attribute selectors
    Not separate stylesheets or duplicated rule sets: :root[data-theme="dark"] { --surface: oklch(15% 0 0); }. Every component references tokens, never hardcoded colors.
  • 3 · Derive
    Tinted/translucent surfaces from base tokens
    Use color-mix() instead of a second hardcoded color per surface — badges, glows, hover states, buttons all follow the base hue automatically.
  • 4 · Prefer
    A modern color space — oklch
    Over hex/rgb when the source allows it. Makes "same hue, different lightness" a one-value L-channel change — exactly the transform a reskin needs.
  • 5 · Wire
    The toggle to a single DOM attribute
    document.documentElement.dataset.theme = 'dark'. Because every color is a variable reference, the whole palette recomputes — no other JS, no re-render logic.
  • 6 · Verify
    Both themes render correctly, same session
    Toggle back and forth without a page reload; confirm all surfaces update, not just the obvious ones — badges, glows, disabled states, borders.
Decision Point · did the pattern hold?
If a "new brand" request still requires editing component files, then the token boundary wasn't drawn correctly — go back and pull the hardcoded value into a token. If markup or interaction logic needs to change to make a retheme work, then the reskin has stopped being a reskin.
Gotchas & Principles
  • Keep theme tokens isolated from component CSS — a tokens file/section separate from component rules — so future re-themes never touch component code again.
  • Don't hardcode a second literal color for "the tinted version of X" — derive it via color-mix() or it drifts out of sync on the next brand change.
  • A well-drawn token boundary turns a brand reskin into a data change (new token values), not a code refactor.
