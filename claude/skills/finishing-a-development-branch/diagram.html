finishing-a-development-branch

Ship, Keep, or Discard

Verify tests → present exactly three options → execute → clean up. Integration into dev always goes through a PR, owned by the github bundle's ship operation.

STEP 1

Verify Tests

Run the project's test suite. Failing tests stop everything — no options are offered until the suite is green.

STEP 2

Determine Base Branch

git merge-base against dev, then main/master — or confirm with the user.

STEP 3

Present Exactly 3 Options

Concise, structured, no added explanation.

▼
Option 1

Ship to dev

Delegate to the github bundle's ship operation: branch naming, push, PR against dev, merge commit, branch cleanup.

→ then remove worktree
Option 2

Keep as-is

Branch and worktree preserved untouched — the user handles integration later.

→ worktree kept
Option 3

Discard

Permanently deletes branch, commits, and worktree. Requires the user to type discard exactly.

→ then remove worktree
Option 1 delegates to github › ship (/ship) — the single owned path for landing work on dev.
Removed by design: local merges into the base branch. They bypass PR review, branch protection, and the merge-commit history convention. Never push directly to dev or main.
