React Virtualization with jsdom Measurement

react-window · TanStack Virtual · large lists under jsdom · behavior vs. pixels

When to Use
Use when implementing or testing row/item virtualization (react-window, TanStack Virtual) for large lists (1,000–10k+ rows) in a React app whose test suite runs under jsdom rather than a real browser. jsdom performs no real layout, so virtualization code that depends on measured element sizes needs a deliberate testing strategy — this skill is that strategy.
five-step method, one virtualized list
Method · in order
  • 1 · Seed
    Stub non-zero layout in test setup
    jsdom's getBoundingClientRect() returns all-zero rects by default. Stub it in vitest.setup.ts to return a fixed non-zero height (e.g. height=640) so measureElement-based dynamic sizing is unblocked.
  • 2 · Render
    Fixed-height row renderer, absolute positioning
    Mount only the visible viewport (e.g. ~25 rows for 10k+) while the full dataset stays in memory. Position each row with inline top: pos * rowHeight. Row container must be position: absolute, not relative.
  • 3 · Assert
    Test DOM shape, not pixels
    Assert what jsdom can represent correctly: rendered row count matches the viewport window, scroll-sizer height equals total_rows * rowHeight, text content matches the DOM.
  • 4 · Scope
    Scope assertions to a specific container
    When multiple layout variants (grid vs. card-list) render in the same test DOM, scope queries to the right container — otherwise getByText/getByRole fail with "multiple elements found".
  • 5 · Verify
    Confirm visually in a real browser
    jsdom tests prove behavior and content — not pixel-perfect visual layout. Confirm smooth scrolling and correct visual virtualization separately in a real browser.
stub rect→ render window→ assert shape→ scope query→ browser check
jsdom Proves
Content & behavior
Right rows exist, right count, right total scroll height, correct text content — all DOM-shape facts jsdom can represent without real layout.
Safe to trust
A passing jsdom suite means the windowing logic and data plumbing are correct.
jsdom Cannot Prove
Pixel-perfect visual layout
Smooth scrolling and correct visual virtualization must be confirmed separately in a real browser — jsdom performs no real layout pass.
Never assume from green tests
A passing jsdom suite does not mean the list looks right on screen.
Inputs → Outputs
Input
getBoundingClientRect stub in test setup, row renderer with position:absolute + inline top offset, full dataset in memory.
Process
Windowed rendering (~25 mounted rows for 10k+ items) driven by scroll position, measured or estimated row height.
Output
jsdom suite verifying row count, scroll-sizer height, and content — plus a manual real-browser check of visual correctness.
Gotchas
  • display:none elements measure 0 height in jsdom (unlike real browsers) — a virtualized tree mounted inside a hidden container (e.g. an inactive tab) won't get real measurements and can silently defeat windowing logic without an obvious test failure.
  • The getBoundingClientRect stub returns the same height for every element — expanded/dynamic-height content still lands fully in the DOM in jsdom even though it wouldn't visually grow the same way in a browser. Text-content assertions stay valid; don't infer real visual row-growth from jsdom behavior.
  • position:relative on the row wrapper is a common bug that only shows up visually (rows overlap/misplace) — it will not fail any content-based jsdom test, so it must be checked by eye in a real browser, not assumed safe because tests pass.
  • Dynamic measureElement sizing requires the getBoundingClientRect shim to exist in test setup ahead of time — without it, tests either fail or force a fallback to fixed estimateSize, which doesn't exercise the same code path as production.
