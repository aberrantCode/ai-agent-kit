Deploy Idempotency Two-Pass Gate

run the apply twice · require pass 2 = zero changes · catch non-idempotent tasks before they corrupt future deploys

When to Use
Use before declaring any live infrastructure apply — Ansible playbook, Terraform plan/apply, Docker Compose stack, or any tool that should converge to a stable state — "done."
Any live apply
Before declaring an apply successful, regardless of the tool — the gate is tool-agnostic convergence proof.
Unmerged feature branch → live infra
Especially critical here: the two-pass run is the only chance to catch idempotency bugs before the code reaches dev.
Hard-stop before smoke tests
Convergence bugs get caught early instead of being masked by "it looks like it worked."
two passes, one non-negotiable gate
Method · sequential steps
  • 1 · lock
    Acquire a coordination lock
    Before pass 1, if other agents/processes might apply concurrently — shared rendering tasks (dashboards, compose stacks) will falsely appear non-idempotent if two applies race.
  • 2 · pass 1
    Run pass 1 — expect changed > 0
    Normal on first deploy or after a real change. Capture the full PLAY RECAP (or equivalent apply summary) in the evidence log.
  • 3 · pass 2
    Run pass 2 immediately after — require changed = 0
    Same playbook/command, same scope. Capture this PLAY RECAP too — both recaps go in the evidence log together, not just the final one.
  • 4 · isolate
    Isolate the signal if pass 2 is not clean
    Don't auto-fail on any nonzero changed: filter to target tasks/resources and verify only those went quiet; document unrelated churn as a pre-existing flap.
  • 5 · settle
    First-time deploys may need one settle pass
    Cross-service artifacts (Authentik app binding, dashboard catalog hash) can only reconcile once the resource exists. Run a third diagnostic pass — monotonic decline (6 → 2 → 0) proves convergence; persistent/fluctuating changes prove broken non-determinism.
  • 6 · triage
    Distinguish expected slowness from errors
    First-run install latency / image pulls are expected on pass 1, not failures. Stop and escalate only on a non-zero return code in pass 2, or non-converging changes.
  • 7 · gate
    Treat the gate as non-negotiable and cost-bearing
    A failing pass 2 demands a root-cause fix in the underlying task/template — never a skip-and-retry, never a silenced check.
lock→ pass 1 (changed>0)→ pass 2 (changed=0?)→ gate: pass / root-cause fix
Decision Points · pass 2 is not clean
If unrelated tasks are churning
e.g. a different dashboard's emby.json re-copying, or an unrelated catalog re-rendering → document as a pre-existing flap, not a fail of your own change.
If your own target tasks are churning
Non-zero changed confined to your target resources on pass 2 (with no settle justification) → hard fail, root-cause the template before proceeding.
First-time deploy, third pass declines monotonically
6 → 2 → 0 changed across passes → genuine one-time settle, not a bug. Proceed.
First-time deploy, third pass persists/fluctuates
Changes don't converge → broken/non-deterministic template. Root-cause it, don't skip it.
Gotchas
  • Common false non-idempotency sources: pipx inject heuristics, handlers firing on every run instead of only on real change, dashboard/catalog render tasks hashing inconsistently.
  • Concurrent agents/processes touching shared rendering tasks can make an otherwise-idempotent task look broken — this is why the coordination lock and unrelated-churn isolation matter.
  • Don't confuse a legitimate one-time settle with a real bug — the distinguishing signal is monotonic convergence across a third pass, not just "pass 2 wasn't zero."
  • Skipping the second pass, or accepting changed > 0 on faith, defeats the entire point of the gate — every exception must be documented as a known pre-existing flap, never silently ignored.
