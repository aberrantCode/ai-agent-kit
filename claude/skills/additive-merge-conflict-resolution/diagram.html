Additive Merge Conflict Resolution

recognize false-positive conflicts · resolve as union-keep-both · not a semantic merge

Core Insight · false signal, not a real conflict
When both branches only append or insert new, independent content near the same anchor point — never editing the same existing line — git's line-based diff can't tell the insertions are unrelated. It flags a conflict even though the correct resolution is obvious to a human: keep both.
recognize the shape → resolve as union, not semantics
When to Use · trigger shapes
Registry / handler additions
Two feature branches each register a new scanner, probe, or handler at the same insertion point in a shared registry file.
Append-only doc conflicts
git stash pop or rebase conflicts on progress logs, learnings intake files, or backlog notes where each side added a distinct, valid entry.
Strictly additive edit pattern
The file's edit pattern is new blocks appended or inserted — never in-place modification of existing content.
Method · six-step resolution pipeline
  • 1 · diagnose
    Additive or substantive?
    Read both sides of the conflict. Brand-new, logically independent content near the same anchor = false-positive. Editing the same existing line = real conflict, out of scope for this skill.
  • 2 · resolve
    Keep both sides — never pick a "winner"
    Remove <<<<<<< / ======= / >>>>>>> markers and retain both hunks, preserving each hunk's original internal order.
  • 3 · order
    Append-only logs: order by convention
    Keep both appended blocks, separated by a blank line, ordered by recency (newest first) to match the file's established reverse-chronological convention.
  • 4 · holding-pen
    Stash-pop on intake/holding-pen docs
    Both the stashed version and current HEAD are distinct, valid entries. Keep both; drop only your own session's stash reference; leave no scratch artifacts behind.
  • 5 · verify
    Confirm the union is correct
    Run the test suite for code registries, or visually confirm both entries are present and well-formed for logs/docs — before committing the resolution.
  • 6 · document
    State the merge logic in the commit message
    Say explicitly this was an additive/false-positive conflict and both sides were kept — so git blame/bisect readers understand the resolution wasn't arbitrary.
diagnose→ keep-both→ order→ verify→ document
Decision Point · additive vs. substantive
Additive (this skill applies)
Each side inserts distinct, independent content at/near the same anchor. Union of both hunks is always correct — resolve fast.
Substantive (out of scope)
Both sides edit the same existing line(s) with conflicting intent. Requires real semantic judgment — not a keep-both operation.
Gotchas
  • Don't reach for a merge tool's "ours"/"theirs" shortcut — both sides are correct; picking one silently loses data.
  • Order matters for chronological logs — appending the older entry above the newer one silently breaks convention even with no data loss.
  • This pattern recurs constantly on shared append-only files touched by concurrent sessions/PRs — recognize the shape fast rather than re-deriving it each time.
  • Always re-run tests after registry-style resolutions — "both blocks present" doesn't guarantee the combined registration is still valid (e.g., duplicate keys).
