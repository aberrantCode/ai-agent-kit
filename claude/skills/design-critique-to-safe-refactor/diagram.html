Design Critique to Safe Refactor

critique/UX finding → code change · contract-preserving · zero silent regressions

Purpose
Converts a design critique, redesign request, or UX audit finding into real code changes on a working tool — without silently breaking it. Treats tests and client-side DOM hooks as the interface contract to preserve, and gates auth/security-sensitive side effects with explicit review before merge.
Applying critique findings
A design-system audit or your own critique becomes code changes on an existing UI.
Redesigning a live workflow
Feature already has real usage, client-side wiring, or an existing test suite.
Auth/security side effects
Refactor touches auth or another core path as a side effect of the design change.
seven-step method, contract-first
Method
  • 1
    Read the test contracts first
    Tests are the source-of-truth interface contract. Know which assertions break under each proposed change before making it.
  • 2
    Snapshot diffs need deliberate regeneration
    A diff isn't automatically a bug. Update snapshots only for changes you intended; unexpected diffs mean investigate, not blindly regenerate.
  • 3
    Classify every test failure
    Intentional removal (critique says so) → update the test. Not called for → it's a regression, fix the code.
  • 4
    Preserve every existing DOM hook
    Read the existing client JS first, find every class/id/data-* hook it queries. Restructure everything else freely; prefer CSS-only mechanisms over JS changes.
  • 5
    Orchestrate, don't reimplement
    For workflow redesigns, build the new surface over existing proven endpoints/data paths so it inherits write-path safety guarantees.
  • 6
    Gate auth-touching changes explicitly
    Run code-reviewer + security-reviewer before merge. Verify the fix on every path (e.g. forward-auth and OIDC), test guard assertions directly, record known tradeoffs.
  • 7
    Split multi-implementation audits by lens
    Parallel subagents per lens (tokens, docs, visuals). Ground image-blind auditors with a written UI inventory. Sequence fixes so lower-numbered findings unblock higher ones.

Gotchas
  • A green test suite after a redesign doesn't prove nothing broke — confirm that the tests you expected to change actually changed, and nothing else did.
  • "The design looks right" is not the same claim as "the client-side hooks still resolve" — verify both.
  • Dual-path checks (e.g. two auth mechanisms) are the classic spot a fix looks complete but only covers one path — enumerate every path explicitly.
  • Image-blind subagents fabricate visual detail from vague prompts — always ground them with a written inventory first.
