Spec Consistency & Doc Refactoring Pattern

spec-vs-reality drift · structural doc repair · atomic, scope-limited fixes

Purpose
Resolve inconsistencies between design/spec documents and deployed reality, or repair structural drift in large markdown docs (mangled backlogs, misaligned specs, redundant catalog fields) — atomic, scope-limited fixes that preserve intent and prose, not a full re-architecture.
ten-step method
Method
  • 1
    Scope to exactly the finding
    Don't expand a consistency fix into a re-architecture, even the "lighter half" of a decision made elsewhere.
  • 2
    Classify before touching anything
    Doc-only (zero live/runtime effect) vs needs a code/config change too — don't conflate in one commit.
  • 3
    Prefer line-oriented transforms
    Avoid YAML parse-and-reserialize round-trips — they silently destroy comments and exact formatting.
  • 4
    Use standard markdown structures
    Footnotes, examples, headings signal relationships ("this supersedes that") without a bespoke syntax.
  • 5
    Repair mangled docs atomically
    Anchor-based joining for a bijection old↔new; fence-aware line wrapping (skip tables, headings, rules, code spans).
  • 6
    Verify, don't eyeball
    Word-multiset diff proves no prose was lost/altered; confirm every cross-repo anchor reference still resolves.
  • 7
    Derive, don't duplicate
    If a field is computable from another canonical field, don't maintain both — usually the drift's root cause.
  • 8
    Update every SoT mirror together
    Central spec, README/SKILL docstrings, the enforcing lint rule, and a migration script — one coherent change.
  • 9
    Prove new lint rules negatively
    Show the rule fails on the old/bad shape and passes after migration, before shipping it enforced.
  • 10
    Prefer atomic, single-commit changes
    Trivially revertible; pair with a design-review step when the fix touches a canonical decision.
Decision Point · classify the fix
Doc-only
Zero live/runtime effect (e.g. a commented-out placeholder) — moves faster, less review risk.
Code/config required
Live behavior change — treat as a separate, heavier-weight change; don't conflate with doc-only edits.
Gotchas
  • A commented-out placeholder is easy to mistake for "doc-only" — check for a live behavior change before scoping the PR.
  • YAML round-tripping is the most common way to accidentally destroy formatting/comments — default to line-oriented edits.
  • Redundant catalog fields usually mean a missing "derive, don't duplicate" step — fixing the symptom just delays the next drift.
  • Don't skip the word-multiset diff on large structural repairs — cheapest way to catch accidental prose loss.
