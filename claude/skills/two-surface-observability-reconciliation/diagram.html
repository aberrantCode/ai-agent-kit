Two-Surface Observability Reconciliation

two blind surfaces · one diagnostic probe · a single deliberate verdict

When to Use
A system's true state can only be seen by combining two observability surfaces that can't see each other — or two candidate sources of truth disagree about what's actually going on. Reconcile them into one verdict instead of trusting either alone.
Surface A · API / tool-level view
In-session tool layer, formal PM framework (specs → plans → tasks), or a container healthcheck. Can look "recent and coherent" while being blind to a stuck downstream stage.
Surface B · backend / infra-level view
Direct SSH/DB query, a living backlog doc, or a live log tail / metric sample. Reveals what the other surface never reports.

four-part reconciliation method
Method
  • probe
    Build a diagnostic probe per surface
    Read each surface in its native format, separately, then compare. E.g. latest-derivation timestamp on each side, or error count vs. throughput.
  • choose SoT
    Pick the authoritative source deliberately
    Formal framework fits greenfield work with a defined completion state; a living backlog fits perpetual infrastructure repos. Report by priority/theme, not % complete, when backlog wins.
  • unstick
    Separate host health from live app state
    A docker ps "unhealthy" can be stale. Confirm real failure via fresh log activity, daemon restart, and live metrics (memory, load, io.pressure) recovering.
  • derive
    Prefer deriving from an existing collector
    E.g. increase(metric_name[1h]) from an exporter that already polls, over standing up a new log-scraper. Validate idempotency of any backfill (second pass = changed=0).
Gotchas
  • A surface that looks "recent and coherent" can still be blind to a stuck downstream stage.
  • Don't average or merge two disagreeing surfaces' numbers — pick the authoritative one and state why.
  • Healthchecks freeze mid-wedge and report stale status — always cross-check with a live log tail or fresh metric.
  • Building new collection infrastructure to fill a gap is a last resort — check for an existing poller/exporter first.
