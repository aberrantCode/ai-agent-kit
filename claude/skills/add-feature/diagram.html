Add Feature Spec

One skill, two workflow styles — a conversational 7-phase interview or a template-first expansion of a single sentence. Both land a thorough spec in /docs/features/.

Project-manager guard: if docs/workflow/SDLC.md + docs/features/template.md + docs/tasks/ exist, stop and invoke project-manager:add-feature (CAP-ID lifecycle) instead.

Phase 0 — Style Selection

/add-feature or a conversational idea → Conversational · /create-feature-spec or one sentence + canonical template → Template-first · ambiguous → AskUserQuestion

Conversational Mode

/add-feature · guided 7-phase interview
  1. Feature identity — name, problem statement, audience
  2. Context & motivation — who asked, priority, background
  3. Codebase scan — find affected files, confirm with user
  4. User stories & personas — measurable acceptance criteria
  5. Technical design — grounded in the Phase 3 scan
  6. Out of scope — the anti-scope-creep section
  7. Risks & open questions — then generate the date-prefixed spec

Template-First Mode

/create-feature-spec · single sentence → canonical template
  1. Initial analysis — read FEATURE_SPECIFICATION.md template
  2. Gather context — scan codebase before asking anything
  3. Clarifying questions — 3-4 strategic, via AskUserQuestion
  4. Initial draft — all REQUIRED sections, substantive content
  5. Iterative refinement — improve sections until approved
  6. Quality checklist — 10-point gate before completion
  7. Save — kebab-case name in /docs/features/
